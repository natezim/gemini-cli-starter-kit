# Report Optimization

## Visual count
- Under 8-10 visuals per page. Each = its own DAX query.
- 20 visuals → 8 = load from 5s to <1s (real-world testing).
- Use separate pages with drillthrough for different content.

## Slicers
- Biggest performance offenders — each generates 2 queries.
- Enable "Add an Apply button" to delay query execution.
- Use dropdown/list slicers over "between" slicers (fewer queries).

Query reduction settings:
```
File → Options → Query Reduction → "Add an Apply button to each slicer"
Optimize ribbon → Optimization presets → "Query reduction"
```

## Bookmarks and hidden visuals
- Hidden visuals behind bookmarks still execute queries unless specifically configured.
- Toggle visibility via bookmark "Data" property set to false.

## Custom visuals
- Run in sandboxed iframes — significant performance overhead.
- Prefer native visuals when possible.
- Each custom visual = additional iframe + query.

## Report page design
- Fixed canvas size (16:9 = 1280×720) for consistent rendering.
- Group related visuals on the same page.
- Use drillthrough pages for detail views instead of cramming everything on one page.
- Tooltip pages: keep them simple (1-2 visuals max).

## DAX query tips for performance
- Aggregations at the visual level: use SUMMARIZECOLUMNS (auto-generated by PBI)
  not manual SUMMARIZE + ADDCOLUMNS.
- Avoid measures that reference other measures in long chains (each link = re-evaluation).
- Format strings in the model, not in DAX (FORMAT function prevents folding).

## Performance Analyzer
Built-in tool in Power BI Desktop:
```
View → Performance Analyzer → Start recording → interact → Stop
```
Shows per-visual: DAX query time, visual rendering time, other processing.
If DAX query time dominates: optimize the measure.
If visual rendering dominates: reduce marks/data points or simplify the visual.
